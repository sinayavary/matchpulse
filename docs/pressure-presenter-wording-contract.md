# Pressure Presenter Wording Contract

## Purpose

This document defines safe presenter wording for future pressure hints.

It is a contract for later user-facing text, not an implementation.

Pressure remains internal-only for now, and presenter exposure stays blocked until this wording contract is approved and a later implementation phase is explicitly opened.

## Current Boundary

- SignalCore can internally emit `PRESSURE_HINT_AVAILABLE`.
- The internal route can return pressure only with `includePressure=true`.
- The presenter does not yet request or expose pressure.
- Public API, frontend, and Telegram remain blocked from pressure exposure.

## Approved Presenter Meaning

Pressure may only mean:

```text
A rule-based pressure hint derived from stored TxLINE score snapshot fields.
```

Pressure must not mean:

```text
prediction
probability
confidence
betting recommendation
winner estimate
next-goal estimate
market edge
team quality ranking
```

## Approved Labels

Allowed labels:

```text
No pressure hint
Low pressure hint
Medium pressure hint
High pressure hint
Limited pressure hint
```

Do not use:

```text
Very high pressure
Guaranteed pressure
Goal pressure
Scoring pressure
Winning pressure
Bet signal
Strong pick
Edge detected
```

## Approved Short Messages

Allowed:

```text
Rule-based pressure hint is available from stored score data.
Current pressure hint: low.
Current pressure hint: medium.
Current pressure hint: high.
Pressure hint is limited by available stored score data.
This pressure hint is based on sparse score-state and possessionType evidence.
This is not a prediction, probability, or betting recommendation.
```

## Forbidden Wording

Explicitly forbid:

```text
Team X is likely to score.
Team X is likely to win.
The next goal is coming.
This is a high-confidence signal.
The model predicts the winner.
This is a betting edge.
This is a sharp move.
Take this pick.
Recommended bet.
Guaranteed pressure.
```

Also forbid these exact words in pressure presenter copy unless part of the safe disclaimer:

```text
probability
confidence
prediction
recommendation
recommended_bet
bet
wager
stake
expected_value
edge
winner
profit
payout
wallet
deposit
```

## Approved Future Presenter Shape

For a later implementation phase, the presenter may expose a compact optional field:

```ts
pressure_hint?: {
  label: "No pressure hint" | "Low pressure hint" | "Medium pressure hint" | "High pressure hint" | "Limited pressure hint";
  level: "none" | "low" | "medium" | "high";
  source: "stored_scores_snapshot";
  evidence_count: number;
  limitations: string[];
  safe_scope_note: string;
}
```

Safe scope note must be:

```text
This pressure hint is rule-based and based on available stored score data. It is not a prediction, probability, or betting recommendation.
```

Important:

- This is only a proposed future presenter shape.
- Do not implement it in this phase.
- Do not include raw payload.
- Do not include debug lineage.
- Do not include formulas.
- Do not include `pressure_score` in presenter by default.
- Do not include `primary_side` in presenter by default.
- Do not include team-specific future outcome language.

## Future Integration Gate

Before presenter integration:

- `agent-presenter-v0.ts` must intentionally opt into `includePressure`.
- Pressure text must be generated by a small safe helper.
- The helper must be unit tested.
- `assertNoForbiddenSignalFields` must pass.
- Public API must remain blocked until separate review.
- Frontend must remain blocked until separate review.
- No betting/prediction/probability wording.
- Presenter must not expose pressure score by default.

## Proposed Future Phases

- Phase 1H - Add pure presenter pressure mapping helper
- Phase 1I - Wire presenter pressure behind includePressure
- Phase 1J - Internal presenter route smoke/test
- Phase 1K - Frontend pressure card contract

Clarify:

- Phase 1H should be pure/helper only.
- Phase 1I may modify `agent-presenter-v0.ts`.
- Phase 1J may modify internal presenter route only.
- Phase 1K is frontend contract only, not full UI implementation.

## Final Decision Table

| Decision | Status | Reason |
| --- | --- | --- |
| Presenter pressure wording | Approved in contract | safe labels/messages defined |
| Presenter pressure implementation | Blocked | needs helper + tests |
| Public API exposure | Blocked | requires separate review |
| Frontend exposure | Blocked | requires display contract |
| Probability wording | Blocked | not calibrated and unsafe |
| Betting wording | Blocked | outside product safety boundary |
