# MatchPulse — Demo Script

A walkthrough for judges, reviewers, and anyone evaluating the MatchPulse platform.

---

## The Idea (One Paragraph)

MatchPulse is a real-time sports data intelligence platform that ingests structured match data from the TxLINE oracle on Solana, persists it in a PostgreSQL database, assembles a canonical match state, then runs data-quality analysis (SignalCore) and generates a natural-language briefing (Agent Presenter) — all surfaced through a safe public demo bridge to a frontend that shows exactly what data is available, how fresh it is, and what's missing. The system tells you *"what we know about this match"* without making predictions, recommendations, or any betting-related claims.

## The Problem

Sports data consumers — analysts, traders, media — need to quickly assess the **quality and completeness** of live match data. Is the scoreboard reliable? Are odds available? How fresh is the data? Existing tools either drown users in raw feeds or hide quality issues behind polished interfaces. There's no layer that says: *"Here's what we have, here's what we don't, and here's how confident we are in what we see."*

## Why TxLINE

TxLINE is a Solana-based oracle that publishes structured sports data (fixtures, scores, odds) as on-chain events. Using TxLINE as the data source provides:
- **Verifiable provenance** — data comes from a known on-chain source, not a scraped feed.
- **Standardized schemas** — TxLINE defines clear data formats for fixtures, scores, and odds.
- **Real-time delivery** — data is published as it happens, not polled from a REST API.

MatchPulse consumes TxLINE events, normalizes them into a canonical format, and persists them for analysis.

## Why Neon DB

Neon is a serverless PostgreSQL provider. Using Neon gives the project:
- **Reliable persistence** — ingested data survives server restarts and can be queried historically.
- **Prisma ORM integration** — type-safe database access with migrations.
- **Serverless scaling** — no infrastructure management for the demo.

## What SignalCore Does

SignalCore is the data-quality analysis engine. Given a canonical match state, it produces a set of categorized signals:
- **Info signals** (e.g., `DATA_READY`) — data is available and usable.
- **Warning signals** (e.g., `ODDS_MISSING`, `SCOREBOARD_MISSING`) — expected data is absent.
- **Critical signals** (e.g., `DATA_STALE`) — data exists but may be outdated.

SignalCore does not predict outcomes, calculate probabilities, or recommend actions. It only reports data quality.

## What Agent Presenter Does

Agent Presenter takes the SignalCore output and the canonical match state and generates a structured brief:
- A **headline** summarizing the match status.
- An **overview** paragraph describing the data landscape.
- Lists of **available data** and **missing data**.
- A **freshness note** indicating data age.
- A **safe scope note** explicitly stating what the system does and does not do.

The brief is informational, not advisory.

## What the Public Demo Bridge Does

The public demo bridge is a safety layer between the backend pipeline and the frontend:
- It serves **only two hardcoded fixtures** — no arbitrary fixture IDs.
- It **sanitizes all output** through `assertNoForbiddenSignalFields` to ensure no betting-related content leaks.
- It stamps every response with `meta.source: "demo-bridge"` and `meta.mode: "public-demo"`.
- It returns a safe `no_data` response for any fixture not on the allowlist — no error details, no stack traces.

The frontend calls **only** demo bridge routes. Internal routes (`/api/internal/*`) are never accessible from the browser.

## What the UI Shows

The `/demo` page renders five panels for each selected match:

1. **Match Intelligence Card** — top-level readiness summary (ready/partial/empty) with key flags.
2. **Agent Brief** — the natural-language briefing generated by Agent Presenter.
3. **Signal Feed** — all data-quality signals, color-coded by severity.
4. **Data Quality Panel** — structured breakdown showing what data is present and what's missing.
5. **Raw JSON Toggle** — the full API response, available for transparency and verification.

## Live Demo Flow

Follow these steps to demonstrate the platform:

### 1. Start the Backend

```powershell
cd D:\money\matchpulse_repo
pnpm.cmd --filter @matchpulse/api dev
```

Wait for the server to start (logs will show `Server listening at http://0.0.0.0:4000`).

### 2. Verify API Health

```powershell
curl.exe http://localhost:4000/api/health
```

Expected: a JSON response confirming the API is running.

### 3. Verify the Public Demo Bridge

```powershell
curl.exe http://localhost:4000/api/demo/matches
```

Expected: two fixture cards with `source: "demo-bridge"` and `mode: "public-demo"`.

### 4. Start the Frontend (separate terminal)

```powershell
cd D:\money\matchpulse_repo
pnpm.cmd --filter @matchpulse/web dev
```

Wait for Next.js to start (logs will show `Local: http://localhost:3000`).

### 5. Open the Demo Page

Navigate to: **http://localhost:3000/demo**

The page should load and display two match options in the selector.

### 6. Select "Slovenia vs Cyprus"

Click the Slovenia vs Cyprus match in the selector.

**Explain:** This fixture has scoreboard data available (score: 1–1) but odds data is missing. The system correctly reports this via:
- `DATA_READY` signal — fixture and score data were ingested successfully.
- `ODDS_MISSING` signal — no odds data was found for this fixture.
- The Data Quality Panel will show `has_scoreboard: true` and `has_odds: false`.
- The Agent Brief will mention what data is available and what's missing.

### 7. Select "Mexico vs South Korea"

Click the Mexico vs South Korea match in the selector.

**Explain:** This fixture has odds data available but scoreboard data is missing. The system reports:
- `ODDS_AVAILABLE` signal — odds snapshots were ingested.
- `SCOREBOARD_MISSING` signal — no score data was found.
- The Data Quality Panel will show `has_odds: true` and `has_scoreboard: false`.
- The Agent Brief will reflect this complementary data landscape.

### 8. Show Raw JSON

Toggle the "Raw JSON" panel for either match.

**Explain:** The raw API response is shown in full, demonstrating transparency. Notice:
- `meta.source: "demo-bridge"` — confirms the response came through the safe bridge.
- `meta.mode: "public-demo"` — confirms this is a demo context.
- `data.readiness.display_ready` — boolean indicating the fixture has enough data to display.
- Signal fields contain only `{ type, severity, title, message }` — sanitized, no extra data.

### 9. Explain Safety Boundaries

Close the demo by emphasizing:

- **No predictions.** MatchPulse does not predict match outcomes, scores, or any future events.
- **No recommendations.** The system does not suggest any actions.
- **No betting mechanics.** There are no wagers, bet buttons, bookmaker links, or wallet connections.
- **No probability output.** No confidence scores, expected values, or edge calculations.
- **Hardcoded fixtures.** The demo only serves two specific fixtures — no arbitrary data access.
- **Sanitized output.** All signals pass through `assertNoForbiddenSignalFields`.
- **Separation of concerns.** The frontend calls only `/api/demo/*` routes. Internal routes are never exposed.

### 10. Verify Mock Endpoints Are Separate

```powershell
curl.exe http://localhost:4000/api/matches
```

Expected: mock-backed response with `source: "mock"` and `mode: "mock"` — completely independent of the demo pipeline.
