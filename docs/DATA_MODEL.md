# MatchPulse Data Model

## 1. Purpose

This file defines the first version of the database model for SignalCore Agent and MatchPulse App.

Recommended stack:

- Neon.tech managed PostgreSQL
- Prisma ORM starting in Phase 10
- UUID primary keys for internal records
- TxLINE fixture IDs stored as strings

Optional later infrastructure:

- Upstash Redis for cache, queue, rate limiting, worker health, or latest match state cache when there is a concrete need
- Redis is not required for Phase 9A or Phase 9B

---

## 2. Core Tables

## 2.1 fixtures

Stores match metadata.

| Field | Type | Notes |
|---|---|---|
| id | uuid | internal primary key |
| fixture_id | text | TxLINE fixture ID, unique |
| competition | text | e.g. World Cup |
| stage | text | group stage, round of 32, etc. |
| sport | text | soccer |
| start_time_utc | timestamptz | match start time |
| home_team | text | participant 1 / home designation |
| away_team | text | participant 2 / away designation |
| status | text | NS, H1, HT, H2, F, etc. |
| raw | jsonb | raw TxLINE fixture payload |
| created_at | timestamptz | default now |
| updated_at | timestamptz | default now |

Indexes:

- unique `(fixture_id)`
- index `(start_time_utc)`
- index `(status)`

## 5. MVP Data Rule

Do not store every raw update forever unless needed.

For MVP:

- store latest match state
- store important match events
- store odds snapshots only when changed or when linked to a signal
- store every generated signal
- store every scenario update linked to a signal or material state change
- store all replay demo data needed for the demo
