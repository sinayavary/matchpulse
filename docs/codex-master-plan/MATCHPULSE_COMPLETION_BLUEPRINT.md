# MatchPulse Completion Blueprint

> Review-only authority. Baseline: `b4f1bf28e3ad05d4c796ac52a1383cd918182842`. This file does not activate a phase or modify the phase queue.

## Product architecture

```text
TxLINE fixtures / optional live transport
  -> validated REST and SSE adapters
  -> supervised catch-up, checkpoints and deduplication
  -> canonical PostgreSQL timeline and odds state
  -> versioned feature assembly and private model adapter
  -> deterministic prediction composition
  -> labels, replay, evaluation and calibration
  -> versioned public-safe API
  -> Persian/English RTL/LTR web + watchlist/Telegram
  -> Docker/local release, CI and submission evidence
```

Provider credentials and raw payloads stay backend-only. Canonical state is deterministic and replayable. Private weights, coefficients, thresholds and specialist contributions never enter Git or public contracts. Public outputs contain only sanitized sports-intelligence scenarios, quality/freshness and accurate proof state. The product has no wagering, stake, payout, profit, expected-value, trading or ordinary-user wallet feature.

## Phase sequence

1. `10F-C`: TxLINE lifecycle and regression closure.
2. `10G-A/B`: stream supervision, catch-up, checkpoints and deduplication.
3. `10H-A/B/C`: review-only composition pack when separately activated, feature assembly and private model adapter.
4. `10N-A`: offline proof-state validation and public redaction.
5. `10E-D` and `10G-C`: odds and canonical persistence on explicitly permitted isolated local PostgreSQL.
6. `10I`: persisted prediction runtime orchestration.
7. `10J-A/B/C`: temporal labels, evaluation, replay and calibration.
8. `10K`: versioned public-safe API.
9. `10L`: complete Persian/English responsive RTL/LTR web application.
10. `10M`: watchlist and Telegram adapter with local fake evidence.
11. `10O-A/B/C`: security, reliability, recovery and Docker/local reproducibility.
12. `10P`: CI, seeded end-to-end evidence and competition submission release.

`10N-B` Solana/on-chain work is deferred and is not a dependency of completion.

## Cross-phase invariants

- One active phase, exact baseline, exact allowlist, verified payload hashes and exact-path staging.
- Local or ephemeral PostgreSQL migrations only when the active phase explicitly permits them.
- No shared/remote database mutation, real service access, deployment or other remote mutation without separate authorization.
- Missing external credentials block only live verification; fixtures, fakes and contract tests remain mandatory.
- Deterministic UTC ordering, idempotency, bounded retry/concurrency, safe errors, secret protection, data integrity and public redaction.
- `ACTIVE_PHASE.json` and `PHASE_QUEUE.json` remain unchanged except through a separately approved exact transition.

## Completion standard

All applicable phases and the quality matrix pass; local migrations and the seeded end-to-end scenario succeed; the API and both web directions work; configurable external adapters have honest offline/live evidence; Docker/local startup is reproducible; CI, documentation and submission artifacts are complete; and no known P0/P1 defect or critical/high security finding remains.
