This document is the final repository-controlled acceptance contract for the competition release. It does not activate production phases and does not claim provider availability, model calibration, financial suitability, or cryptographic verification of derived predictions.

## Final acceptance surface

The release is acceptable only when all of the following remain true:

1. `/competition` renders the permanent public-safe competition prediction families.
2. Prediction output and market analysis remain visually and semantically separate.
3. Deterministic replay exposes the ordered opening, pressure-shift, and terminal checkpoints.
4. Live or stored mode uses the existing versioned public competition prediction endpoint and falls back explicitly to approved replay data when necessary.
5. Loading, unavailable, no-data, stale, partial, replay-fallback, and terminal states remain bounded and understandable.
6. No wallet, payment, private token, provider credential, migration, database mutation, or external provider access is required for evaluator replay.
7. Browser props and public API responses exclude provider identities, raw payloads, private assessment identifiers, feature references, formulas, thresholds, weights, component scores, credentials, secrets, internal errors, and proof blobs.
8. The product contains no wagering recommendation, stake, payout, profit, expected-value recommendation, trade execution, or gambling workflow.

## Required deterministic evidence

Final QA must run the exact focused prediction/replay suite, API typecheck, API production build, full API regression, web typecheck, web production build, diff check, and empty Prisma/migration diff listed by the active phase pack.

The Next production build may rewrite `apps/web/next-env.d.ts` and `apps/web/tsconfig.json`. Those two files are build-generated compatibility targets for this QA phase only. They must be restored exactly to their baseline blobs after the successful build and must not appear in the completion commit.

## Evaluator walkthrough

1. Open `/competition?mode=replay&checkpoint=opening-balance` and confirm a first-half 0-0 state with fresh, strong, low-volatility market evidence.
2. Select `pressure-shift` and confirm a second-half 1-1 state with aging freshness, mixed agreement, high volatility, and visibly changed scenario probabilities.
3. Select `terminal-home` and confirm a finished 2-1 state with terminal final-outcome and no-further-goal probabilities while stale market evidence remains visible but unusable.
4. Confirm every checkpoint keeps prediction and market analysis separate.
5. Confirm protected internal fields and prescriptive product language are absent.
6. Confirm live/stored unavailability is explicit and falls back to approved deterministic replay.

The walkthrough is a release acceptance script. Automated tests and builds are the authoritative evidence generated by the phase execution; this document does not fabricate a manual observation result.
